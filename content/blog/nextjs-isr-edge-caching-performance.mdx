    <h2>ISR vs Edge Caching for the Statistics API</h2>

    <!-- TODO: The statistics route at /api/statistics fetches GitHub + LeetCode data.
    Describe your caching decision:

    The route currently uses:
      Cache-Control: public, s-maxage=3600, stale-while-revalidate=7200

    This means:
    - Edge CDN serves cached response for 1 hour without hitting origin
    - After 1 hour, serves stale while fetching fresh in background (up to 2 hours)

    Why not ISR (generateStaticParams + revalidate)?
    - API routes can't use ISR — only page routes
    - For API routes, Cache-Control is the right mechanism

    What was the latency without caching? (GitHub API: ~300ms, LeetCode: ~500ms)
    With edge caching: ~10ms on cache hit.

    Be specific about the measurement if you made one. -->

    <pre><code>

// app/api/statistics/route.ts — the cache header decision
return NextResponse.json(data, {
headers: {
// Cache at edge for 1h; serve stale for 2h while revalidating in background.
// Without this: ~800ms per request (GitHub + LeetCode in parallel).
// With this: ~10ms on cache hit at Vercel's edge.
"Cache-Control": "public, s-maxage=3600, stale-while-revalidate=7200",
},
});
</code></pre>

    <h2>Takeaways</h2>

    <ul>
      <li><strong>Run ANALYZE=true before shipping.</strong> 195kB hidden in a validation library is easy to miss without the treemap.</li>
      <li><strong>Cache-Control on API routes is ISR for API routes.</strong> s-maxage + stale-while-revalidate gives you the same edge-caching behavior as page-level revalidate.</li>
      <li><strong>AVIF is worth enabling but measure cold starts.</strong> next/image handles the fallback automatically; the risk is encode latency on first request.</li>
